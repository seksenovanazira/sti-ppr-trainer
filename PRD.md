# PRD: Обучающий тренажёр для сотрудников (СТИ, ППР, приёмка кабеля)

**Продукт:** Локальное веб-приложение для обучения и аттестации персонала АО «Казахтелеком»
**Версия документа:** 1.0
**Дата:** 03.07.2026
**Статус:** Черновик для внутреннего обсуждения

---

## 1. Обзор

Приложение предназначено для обучения новых и действующих сотрудников нормативам ППР, признакам неисправностей СТИ и правилам приёмки кабельных линий, построенных подрядчиком. Состоит из трёх модулей: квиз-тренажёр, интерактивная схема осмотра объектов, админ-панель с результатами тестирования.

Ключевое архитектурное требование: **приложение работает полностью офлайн**, без серверов, платных API, подписок и внешних баз данных. Один HTML-файл, вся логика — на стороне клиента (браузера).

## 2. Цели

- Сократить время адаптации новых сотрудников на проверках СТИ и ППР
- Стандартизировать знания по приёмке кабельных линий от подрядчиков
- Дать руководителю участка простой способ видеть, кто из сотрудников прошёл тестирование и с каким результатом
- Обойтись без затрат на инфраструктуру (серверы, БД, платные сервисы)

## 3. Целевая аудитория

- Новые сотрудники (монтажники СТИ, электромонтёры, кабельщики-спайщики) — проходят обучение и тестирование
- Инженеры ПТО, мастера участков — проходят тестирование и участвуют в приёмке линий от подрядчиков
- Руководитель участка / наставник — просматривает результаты через админ-панель

## 4. Функциональные требования

### 4.1 Модуль «Квиз-тренажёр»

**Назначение:** проверка знаний по нормативам ППР, признакам неисправностей СТИ, охране труда и правилам приёмки кабеля.

Требования:
- Банк вопросов с категориями: ППР, СТИ, Приёмка кабеля, Охрана труда
- Типы вопросов: один верный ответ, несколько верных, верно/неверно
- Два режима: «Тренировка» (мгновенная обратная связь и объяснение) и «Экзамен» (результат только в конце)
- Фильтр перед стартом: категория, количество вопросов, уровень сложности
- Прогресс-бар и счётчик правильных/неправильных ответов во время прохождения
- Итоговый экран: процент, разбивка по категориям, список ошибок с объяснениями
- Результат каждой попытки сохраняется локально и передаётся в модуль админ-панели (см. 4.3)

**Структура данных вопроса:**
```json
{
  "id": "q001",
  "category": "Приёмка кабеля",
  "difficulty": "базовый",
  "question": "...",
  "type": "single",
  "options": ["...", "...", "...", "..."],
  "correctAnswers": [1],
  "explanation": "..."
}
```

### 4.2 Модуль «Схема осмотра»

**Назначение:** обучение через кликабельную схему объекта — что именно проверять в каждом узле.

Объекты:
1. Смотровой колодец / камера кабельной канализации
2. Кабельный шкаф
3. Кросс / оптический кросс
4. Опора воздушной линии связи
5. Приёмка кабельной линии от подрядчика (процесс, не объект — этапный чек-лист)

Требования:
- SVG-схема объекта с кликабельными маркерами (5–10 узлов на объект)
- По клику — карточка: что проверять, типичные дефекты, ссылка на пункт норматива (плейсхолдер)
- Для приёмки кабеля — раскрывающиеся этапы процесса вместо схемы: документация → визуальный осмотр трассы → вводы и муфты → измерения → акт приёмки
- Переключатель между объектами

### 4.3 Модуль «Админ-панель»

**Назначение:** просмотр результатов тестирования сотрудников руководителем/наставником.

Требования:
- Сводные метрики сверху: сколько сотрудников прошли тест, средний балл, сколько ниже порога прохождения
- Таблица сотрудников: ФИО, должность, дата прохождения, итоговый результат (%), статус (Отлично / Хорошо / Средне / Повторить)
- Клик по строке — детализация результата по категориям (ППР / СТИ / Приёмка кабеля / Охрана труда) в виде прогресс-баров
- Пороговые значения статуса: ≥85% Отлично, 70–84% Хорошо, 50–69% Средне, <50% Повторить

**Демо-данные для прототипа (не реальные сотрудники):**

| Сотрудник | Должность | Результат | Статус |
|---|---|---|---|
| Ким Диана | Мастер участка | 100% | Отлично |
| Серикова Айдана | Монтажник СТИ | 92% | Отлично |
| Абенов Нурлан | Кабельщик-спайщик | 78% | Хорошо |
| Тулегенов Ерлан | Электромонтёр | 68% | Средне |
| Жумагулов Асхат | Инженер ПТО | 45% | Повторить |

**Структура данных результата:**
```json
{
  "employeeName": "...",
  "role": "...",
  "date": "2026-03-15",
  "overallScore": 92,
  "categoryScores": {
    "ППР": 95,
    "СТИ": 90,
    "Приёмка кабеля": 88,
    "Охрана труда": 96
  }
}
```

## 5. Нефункциональные требования

- Работа полностью офлайн, без интернет-зависимостей после первой загрузки файла
- Без сторонних серверов, платных API, подписок, внешних баз данных
- Хранение данных — только `localStorage` браузера или локальный JSON-файл
- Один переносимый HTML-файл, открывается в любом браузере без установки
- Адаптивность под мобильные устройства (используется в поле)
- Без сбора персональных данных за пределы устройства пользователя

## 6. Ограничение архитектуры: хранение данных результатов

Поскольку БД и сервер исключены, у `localStorage` есть принципиальное ограничение: **результаты видны только в том браузере/устройстве, где проходили тест**. Для сведения результатов со всех сотрудников в одну админ-панель есть два пути:

- **Вариант А (проще):** каждое устройство ведёт свою локальную историю; для сводного отчёта — ручной экспорт результатов в CSV/Excel с каждого устройства и объединение вручную
- **Вариант Б (требует обсуждения):** лёгкое общее хранилище (например, общий файл на сетевом диске компании, куда результаты дописываются при каждом прохождении) — технически уже не «чистый локальный JS», требует отдельного решения по доступу к файлу

*Решение по этому пункту не принято — вынесено в открытые вопросы (раздел 9).*

## 7. UX/UI

- Три вкладки верхнего уровня: «Квиз-тренажёр», «Схема осмотра», «Админ-панель»
- Крупные кнопки, минимум текста на экран — расчёт на использование с телефона в поле
- Цветовая индикация статуса (успех/предупреждение/ошибка) — по стандартной семантике (зелёный/жёлтый/красный)
- Прототип интерфейса согласован в чате (интерактивный макет с демо-данными)

## 8. Вне рамок текущей версии

- Многопользовательская синхронизация результатов в реальном времени
- Аутентификация/логин сотрудников
- Push-уведомления и напоминания о необходимости пройти тест
- Генерация PDF-сертификатов о прохождении (можно рассмотреть в следующей версии)

## 9. Открытые вопросы

1. Как сводить результаты тестирования с разных устройств без сервера — ручной экспорт или отдельное решение по общему хранилищу?
2. Какие реальные номера пунктов регламентов и норм Казахтелекома нужно подставить вместо плейсхолдеров?
3. Кто отвечает за наполнение и актуализацию банка вопросов после запуска?
4. Нужен ли порог для допуска к самостоятельной работе (например, не ниже 70% для доступа к приёмке линий)?

## 10. Этапы реализации (предложение)

1. Утвердить банк вопросов и реальные критерии по всем категориям
2. Собрать рабочий прототип (квиз + схема + админ-панель) — единый HTML-файл
3. Пилот на одной группе новых сотрудников
4. Доработка по обратной связи, подключение реального экспорта результатов
5. Тиражирование на остальные участки
